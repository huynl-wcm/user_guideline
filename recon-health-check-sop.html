Standard Operating Procedure

Health-Check Reconciliation

Hệ thống tự đối soát Databricks (nguồn thật) vs bảng Fabric mỗi ngày, cảnh báo khi lệch. Tài liệu: cách đọc alert, cách xử lý, khi nào tự fix vs báo IT.

Kênh cảnh báo: chỉ Telegram (email đã bỏ)
01 — Kiến trúc

Hai tầng đối soát

                 ┌──────────────── 2 TẦNG RECON ────────────────┐
 Databricks ───► │ INGESTION   DBX  vs  Fact_A_STORE_SKU_DAY     │
 (source truth)  │ TRANSFORM   EXPECTED vs Fact_A_MCH5_DAY_BY_TYPE│ ──► recon_result
                 │             EXPECTED vs fact_mch5_day_by_type  │     (append-only)
                 └───────────────────────────────────────────────┘            │
                                                                               ▼
  Chuỗi build (cascade):  Base ──► Bronze ──► Gold       NB_RECON_ALERT ──► 📲 Telegram
  Fact_A_STORE_SKU_DAY  Fact_A_MCH5_DAY_BY_TYPE  fact_mch5_day_by_type
  (TEAM build)          (TA build)               (TA build)
  • Gold build FROM Bronze, Bronze build FROM Base → sửa 1 tầng phải rebuild mọi tầng downstream cùng window.
  • Recon đọc qua shortcut, không copy bảng. recon_result append-only → query luôn lọc run_ts mới nhất per (partition_date, source_side) + whitelist source_side.
  • Lịch: daily [D-90, D-1] + weekly full sweep Y-1 (01/01 năm trước → D-1).
02 — Vòng đời 1 sự cố

detect → alert → remediate → verify

Ví dụ thật (store 1544 / 4181 / 2CBX, 06-03..07):

  1. detect — recon thấy base < DBX
  2. alert — Telegram đẩy NEW
  3. remediate — cascade re-run đúng window
  4. verify — re-run recon → flip OK → alert tự ngừng push
03 — Đọc alert

Mỗi dấu hiệu nghĩa là gì

Dấu hiệuNghĩaHành động
NEWLần đầu thấy ca nàyXử lý theo mục 04
REMINDCa cũ chưa xử lý (nhắc mỗi 3 ngày)Đừng bỏ qua — đang tồn đọng
NOT_READYBuild chưa chạy tới ngày đó (KHÔNG phải lỗi data)Re-run build bình thường, không cần backfill
MISMATCH ít storeStale do restate back-dated (≲30 store hoặc % nhỏ)auto-safe → cascade re-run (mục 04)

source_side cho biết tầng nào lệch: DBX_vs_Fact_A_STORE_SKU_DAY = base; EXPECTED_vs_Fact_A_MCH5_DAY_BY_TYPE = bronze; EXPECTED_vs_fact_mch5_day_by_type = gold.

04 — Runbook xử lý

MISMATCH auto-safe

  1. Copy nguyên alert → chạy /recon-remediate (paste vào) → ra plan cascade + window DATE_FROM/DATE_TO (= min/max ngày bị flag).
  2. Duyệt plan + xác định chủ sở hữu từng tầng (mục 05).
  3. DRY_RUN mỗi notebook trước cell .write: so count window sắp ghi vs hiện tại — hợp lý mới ghi (xem BACKFILL.md).
  4. Re-run đúng thứ tự cascade (tầng lệch + mọi tầng downstream, cùng window).
  5. Re-run recon tầng tương ứng → ngày flip sang OK → alert tự ngừng push.
Guardrails: replaceWhere ghi đè cả ngày (mọi store) — store trong alert chỉ tham khảo, KHÔNG lọc store khi ghi. DRY_RUN trước. Đúng thứ tự cascade. Re-run recon để xác nhận.
05 — Tự fix vs Báo IT

Theo chủ sở hữu từng tầng

Tầng lệchChủCách xử lý
Base Fact_A_STORE_SKU_DAYTeam (ta read-only)BÁO IT re-run ingestion + fix gốc. Ta KHÔNG ghi đè được.
Bronze Fact_A_MCH5_DAY_BY_TYPETaTự re-run với date range (overwrite)
Gold fact_mch5_day_by_typeTaTự re-run với date range (overwrite)
Vì base là của team: nếu ca lệch ở base, IT phải re-land base trước, rồi ta mới re-run bronze→gold cùng window. Ta không tự đóng vòng RESOLVED cho ca base một mình được.

Root-cause base stale (báo IT)

Ingestion ghi incremental replaceWhere DAY ∈ [D-3, D-1] → khi IT restate 1 ngày cũ back-dated trên Databricks, Fabric base ngày đó không bao giờ re-land → stale vĩnh viễn. Fix gốc = mở rộng window re-land hoặc re-pull khi có restate.

Mẫu tin báo IT

[Health-Check Recon] Fact_A_STORE_SKU_DAY stale — nhờ re-run ingestion
Bảng: Fact_A_STORE_SKU_DAY (Fabric base)
Window: FDM='2026-06-01', DAY 2026-06-03..2026-06-07
Triệu chứng: Fabric base < Databricks (revenue+qty) — store 1544/4181/2CBX
Nguyên nhân: restate back-dated ngoài replaceWhere D-3..D-1 → ngày cũ không re-land
Đề nghị: (1) re-pull window trên cho các ngày này;
         (2) fix gốc replaceWhere để re-land khi có restate back-dated.
